Those answers collapse most of the design space. Local-first dev infra, two-way git, real-time multi-actor with agents in the loop, Rust core + TS bindings — that's a specific architecture, and the hard problems are now identifiable. Let me sketch it.The shape: a single Rust daemon (`quarryd`) that exposes WS/HTTP/MCP/git-protocol on top, runs a CRDT engine in the middle, mirrors to a real git repo on the side, and projects to FUSE for the agents that want POSIX semantics. One binary, one local data directory, multiple connected clients on the LAN.

## Concrete tech picks

**Loro over Yjs.** You're Rust-native, Loro is Rust-native, the wasm TS bindings are fast, and Loro's text/list/map CRDTs handle high-frequency edits better than Yjs at agent rates. Bonus: built-in undo manager and time-travel, which you'll want for "show me the version 10 minutes ago" and for the gutter UI.

**gitoxide over `git2-rs`.** Pure Rust, no libgit2 C dependency, much cleaner API, actively maintained (Byron is responsive). Critical for embedded use where you don't want libgit2's threading quirks.

**SQLite via `sqlx`.** Metadata in a single file, WAL mode for concurrent read/write, swappable for Postgres later via the same query layer when you go hosted.

**Content-addressed blobs on local FS.** Same shape as git's `.git/objects` — sharded by hash prefix. Trivially S3-able later by swapping a storage trait.

**`axum` for HTTP/WS/SSE, `fuser` for FUSE.** Standard choices. MCP is just HTTP+SSE so it sits in axum cleanly. FUSE on Linux/Mac only; Windows gets the sync-folder path or nothing in v1.

## Crate boundaries

Carve the workspace so each crate has a small, testable contract — this matters more than usual because agents will be implementing them, and clean contracts are the difference between agents doing useful work and agents producing plausible-looking garbage.

`quarry-store` is the only crate that touches the filesystem or SQLite. Trait-based so storage backends are pluggable. Owns blob CAS, metadata schema, op-log persistence, snapshot storage. No business logic.

`quarry-crdt` wraps Loro, defines the per-document op schema, owns provenance — every op carries `actor_id`, `actor_kind ∈ {human, agent, git_import}`, `timestamp`, optional `transaction_id`. Snapshots and op-log persistence go through `quarry-store`. Draft/publish semantics live here.

`quarry-git` is the bidirectional bridge built on gitoxide. Two operations only: `materialize` (CRDT state → real commits) and `ingest` (external commits → CRDT ops). Treats git as an output device on one side and an input source on the other; never as the source of truth.

`quarry-projection` is read-only derived views over `quarry-store`. FUSE adapter, sync-folder daemon, raw export. Stateless — restart-safe, since all state lives in the store.

`quarry-api` is the axum server: REST + WS subscriptions + MCP server + git smart-HTTP endpoint. Auth, ACLs, rate limiting, transaction coordination.

`quarry-daemon` is the binary that wires it all together. Config, logging, lifecycle.

`quarry-sdk-ts` is hand-written, not generated. The WS subscription ergonomics matter and codegen produces ugly clients. Optionally a thin wasm shim that re-exports Loro types so TS callers can hold real CRDT handles for browser-side editors.

## The two genuinely hard problems

**Two-way git sync.** Outbound is easy: on `publish` or on a schedule, walk the CRDT state, diff against the last materialized tree, write a commit with the actor's identity attached. Use signed commits if you want strong provenance — gitoxide supports this.

Inbound is the hard one. When something pushes to your git endpoint (or you pull from a remote), you have a commit that wasn't generated by quarry. You need to: diff the incoming tree against the last-known tree; translate the diff into a stream of CRDT ops; apply those ops attributed to the commit author with `actor_kind=git_import`; and reconcile against any in-flight live edits. Loro will auto-merge concurrent ops by construction, but you still need a policy for semantically meaningful conflicts (someone live-rewrote the function being imported).

The non-obvious failure mode is amplification: an incoming commit that touches 1000 lines becomes 1000 CRDT ops, which then ship out to every live client. You want "one git commit = one op group" so the CRDT applies it atomically and clients render it as one change. Bake the transaction concept into the op schema from day one.

Identity preservation across the round trip is the other subtle one. If Alice commits via git, that flows in as an op with `actor_id=alice@email`. When quarry later materializes that op into a commit going outbound, the author needs to be Alice, not `quarry-bot`. Make this explicit in the schema or you'll lose authorship every roundtrip.

**Agent edit rate.** A coding agent doing a refactor produces thousands of small edits per second. Two problems compound.

First, broadcasting every keystroke-equivalent to every client saturates the websocket. Solution: agents commit in *transactions* — logical units like "rewrote function foo," not "inserted character at offset 14." The API enforces a transaction model; agents that hold a transaction get exclusive access to the affected ranges, other actors see one coalesced update on commit. This also gives you a clean unit for the git projection.

Second, agent edits without coordination trample human work. The cleanest fix is *intent locks*: an agent declares "editing section X for the next 30 seconds," other agents back off, humans see a soft indicator. Linear's sync engine does something similar with optimistic transactions; worth watching the Artman talks.

Provenance gets interesting at this rate. Per-op attribution is fine. Per-character attribution Proof-style is harder when an agent rewrites a paragraph that previously had mixed authorship — you need a policy for whether the new text is "agent" (factually true) or "agent based on human input" (more useful for the human reviewer). I'd start: attribution per-op, the gutter renders the most recent op's actor, "regenerated by agent based on human draft" is a separate explicit relationship stored in metadata so the UI can render it as a derivation chain rather than flat color.

## Build order for agent-driven work

Since agents do the work and time isn't pressing, build the spine end-to-end before adding features. A sequence that maximizes parallel agent work:

1. `quarry-store` with in-memory + SQLite impls and full trait test coverage. Very tight contract — agent-friendly.
2. `quarry-crdt` wrapping Loro, op schema and provenance, no persistence yet.
3. Wire `quarry-crdt` to `quarry-store` for snapshots and op-log. At this point you have a working in-process collab store.
4. Minimal `quarry-api` (HTTP + WS) plus a tiny TS test client. End-to-end realtime collab now works for two browser tabs editing the same doc.
5. `quarry-git` outbound only. Materialization, identity preservation, signed commits.
6. `quarry-git` inbound. This is where most real design work happens; expect to iterate the op schema once you hit real merge cases.
7. `quarry-projection` FUSE adapter. Worth waiting until 1–6 are stable so you're not debugging FS issues on top of a moving substrate.
8. MCP server inside `quarry-api`. Agents can now use quarry through the standard interface.
9. Draft/publish semantics, annotations on binaries, ephemerals, hit/miss tracking — all layers on top of the now-working substrate.

The big risk in this sequence is committing to an op schema in step 2 that needs to change in step 6. Mitigate by versioning the schema from day one, accepting that v0 op-logs may need migration, and writing the migration tool before you need it.

## Worth pinning down before serious work

A few decisions the architecture above silently assumed; flag if any are wrong:

**Per-developer daemon vs per-team daemon.** "Local" can mean each dev runs their own `quarryd` and they sync via git, or one daemon on the LAN that several devs connect to via WS. The first is simpler and matches the "git is the integration substrate" model nicely. The second is more like Linear's experience but needs an auth model in v1. Which one matches your mental picture?

**Conflict policy when git import collides with live edits.** Three options: imported wins, live wins, surface as a draft side-by-side for human resolution. I'd default to the third because it preserves the most information, but it's a UX-heavy choice.

**MCP tool surface.** What's the minimum verb set for agents? My starting guess: `list`, `read`, `write`, `comment`, `start_draft`, `publish_draft`, `search`. "Search" alone is a whole sub-question — full-text via SQLite FTS5 for v1, semantic later? Or both from day one?

**Document size ceiling.** Loro handles large docs, but you'll want a max and a story for what happens beyond it (split into linked nodes, warn, hard limit). Especially relevant for the "binary files with annotations" path — a 200MB PDF isn't a CRDT, but its annotation layer is.
